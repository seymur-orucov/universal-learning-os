# ADR-0048: v0.3.0 Release Preparation

## Status

Accepted

## Context

Stages 18.0 through 22.1 added optional learner runtime documentation and CLI helpers, automated CLI tests, app architecture planning, the local Studio MVP, and Studio hardening. The repository now needs release documentation and final validation guidance for a future `v0.3.0` tag.

Release documents are not canonical specifications. Canonical behavior remains in `specification/`, `core/`, `commands/`, `skills/`, `domains/`, `templates/`, `tools/ulos-cli/`, and `apps/studio/` as applicable.

## Decision

Prepare `v0.3.0` as a documentation and validation stage only.

Stage 23.0 adds v0.3.0 release notes, a release checklist, release-candidate status, this ADR, release index updates, roadmap updates, validation report notes, changelog updates, and a manual acceptance checklist. It does not tag the release.

## Why v0.3.0 Is Release-Worthy

`v0.3.0` groups several user-visible capabilities after `v0.2.0`:

- Optional learner handoff and snapshot workflows.
- CLI helpers for learner artifact scaffolding and validation.
- Automated CLI tests for generated packs and learner helper guardrails.
- A local Studio MVP for CLI-backed inspection, validation, dry-runs, and optional learner helper workflows.
- Studio hardening for local documentation, smoke checks, health checks, bridge error clarity, and security-boundary copy.

## What Changed Since v0.2.0

- Learner runtime strategy became explicit, optional, snapshot-based, and evidence-first.
- `create-handoff`, `create-snapshot`, and `validate-learner` were added to the CLI.
- Automated CLI tests were added with Node's built-in test runner.
- Studio planning docs defined a local-first CLI-backed control panel.
- Studio MVP implementation added a React/Vite UI and local Node CLI bridge.
- Studio hardening added README guidance, smoke scripts, bridge/server checks, and clearer UI copy.

## Why Studio Remains Local-First

Studio is meant to expose existing repository and CLI workflows to a trusted local user. Keeping it local-first avoids expanding the framework into an account system, hosted service, database-backed product, or ChatGPT API integration. The CLI remains the source of truth for domains, pack validation, pack inspection, generation dry-runs, handoff scaffolds, snapshot scaffolds, and learner artifact validation.

## Why Learner Runtime Remains Optional

Daily learning remains in ChatGPT Projects. Learner handoff and snapshot artifacts support continuity when explicitly requested, but repository updates are not required after every lesson. This preserves the evidence requirements in `specification/LEARNING_LIFECYCLE.md` and `specification/STATE_SPEC.md` and avoids automatic mastery claims from displayed content or chat history.

## Why Generated Pack Contracts Remain Unchanged

`v0.3.0` prepares a release around runtime helpers, testing, and Studio. It does not need new domain content or generated pack contract changes. Keeping standard packs at exactly 25 files and compact packs at exactly 5 files protects existing ChatGPT Project setup flows and validation expectations.

## Non-Goals

- Add a new learning domain.
- Regenerate packs without need.
- Change generated pack file-count contracts.
- Replace ChatGPT Projects as the daily learning runtime.
- Require repository updates after every lesson.
- Implement a heavy runtime learner-state system.
- Import or parse full ChatGPT transcripts.
- Infer mastery automatically from chat history.
- Generate YAML learner-state patches.
- Store sensitive personal data.
- Add authentication, cloud storage, database storage, or ChatGPT API integration.
- Build a full LMS.
- Tag `v0.3.0` before final approval.

## Consequences

- The repository has a clear v0.3.0 release-preparation trail.
- Final release validation can be run from documented commands.
- Future Stage 23.1 and 23.2 work can focus on hardening and tagging instead of discovering release scope.
- Generated pack and learner-state boundaries remain stable.
