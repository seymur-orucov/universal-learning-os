# English for Software Engineers

## Domain Identity

- Domain id: `english`
- Domain title: `English for Software Engineers`
- Domain version: `0.1.0-draft`

## Target Audience

- B1 learners aiming for B2/C1 professional English.
- Software engineers who need English for interviews, meetings, documentation, technical explanations, and remote work.
- Learners who want speaking-first English with grammar accuracy and vocabulary growth.

## Domain Goals

- Improve speaking fluency and confidence.
- Improve grammar accuracy for professional communication.
- Build workplace and interview vocabulary.
- Improve listening comprehension.
- Practice shadowing and pronunciation.
- Develop technical explanation skills.
- Improve writing for emails, chat, PR comments, summaries, and documentation.
- Prepare for software engineering interviews and remote communication.

## Expected Outcomes

- Learner can speak about daily work, projects, blockers, decisions, and tradeoffs.
- Learner can explain technical concepts clearly.
- Learner can answer and ask interview questions.
- Learner can write concise professional messages.
- Learner can understand common workplace and technical discussions.
- Learner can use grammar and vocabulary with increasing accuracy.
- Learner can self-correct common speaking and writing mistakes.

## Supported Tracks

- `english.main`: B1-to-C1 speaking-first professional English track.
- `english.speaking`: focused fluency, pronunciation, and shadowing track.
- `english.interview`: software engineering interview English track.
- `english.workplace`: meetings, status updates, blockers, feedback, and written communication track.
- `english.writing`: emails, chat messages, PR comments, summaries, and documentation track.

## Localization Policy

- Instruction language MAY be Azerbaijani for the initial project preference.
- Target language is English.
- Examples, dialogues, vocabulary, collocations, shadowing scripts, and learner output SHOULD be in English where appropriate.
- Explanations MAY be in Azerbaijani when learner or project preference requires it.
- English terms and phrases SHOULD remain in English.
- Localization behavior MUST follow `specification/LOCALIZATION_SPEC.md`.

## Prerequisites

- Approximate A2/B1 English foundation for the main track.
- No requirement for advanced grammar before starting.
- Software engineering context is useful but not required for general communication parts.

## Non-Goals

- This domain is not a full IELTS/TOEFL certification course at draft stage.
- This domain is not a general school grammar-only course.
- This domain does not contain learner-specific progress.
- This domain does not certify C1 level without evidence.
- This domain does not replace real conversation practice with humans.

# English Syllabus

## Syllabus Overview

This syllabus defines the `english.main` learning path for English for Software Engineers. It maps stages, modules, and concise lesson outlines to learning skills from `domains/english/SKILL_GRAPH.md`.

This file defines the learning path only. It does not contain full lesson bodies, dialogues, listening transcripts, shadowing scripts, exercise banks, learner state, or learner progress.

Lesson completion does not imply mastery. Speaking participation does not imply speaking mastery. Generated dialogues, scripts, or shadowing prompts do not imply mastery. Mastery requires captured, summarized, or reviewed learner output as evidence. Learner progress belongs in learner state, not this syllabus.

# English Practice Rules

## Practice Overview

These rules define how the `english` domain pack maps framework practice types to English for Software Engineers. They support future lessons, speaking practice, listening practice, shadowing, writing review, interview simulations, portfolio work, review sessions, and assessments.

This file does not create full exercises, dialogues, listening transcripts, shadowing scripts, lesson bodies, learner state, schemas, or executable tooling.

Practice output SHOULD follow `templates/session/PRACTICE_OUTPUT_TEMPLATE.md`.

## Practice Principles

- Practice SHOULD target learning skills from `domains/english/SKILL_GRAPH.md`.
- Practice SHOULD follow `core/learning-engine/PRACTICE_MODEL.md`.
- Domain rules MUST NOT weaken `core/mastery-model/EVIDENCE_REQUIREMENTS.md`.
- Learner progress belongs in learner state, not this domain pack.
- Practice output MAY propose learner state updates but MUST NOT silently apply them.
- Speaking participation, passive listening, shadowing repetition, generated scripts, or lesson completion MUST NOT imply mastery.
- English target examples, phrases, collocations, interview answers, and learner outputs SHOULD remain in English where appropriate.

## Practice Type Mapping

### Recall Practice

- Purpose: Strengthen retrieval of grammar rules, vocabulary, collocations, phrase patterns, pronunciation distinctions, and interview structures.
- Suitable learning skills or stages: Stage 2 grammar, Stage 4 vocabulary, Stage 5 pronunciation awareness, Stage 9 interview structures.
- Expected learner output: short answer, definition, example sentence, collocation choice, or phrase substitution.
- Evidence value: supports M1 or M2 and may support M3 when output is produced without seeing the answer first.
- Common failure patterns: recognition without production, memorized definitions without use, wrong phrase context.

### Guided Practice

- Purpose: Build controlled production with prompts, scaffolding, models, or partial sentence structures.
- Suitable learning skills or stages: Stage 1 confidence, Stage 2 grammar, Stage 3 status updates, Stage 6 technical explanation.
- Expected learner output: guided spoken answer, controlled written response, corrected sentence, or scaffolded explanation.
- Evidence value: supports M3 when feedback identifies what the learner can do with support.
- Common failure patterns: repeating model text without adaptation, overreliance on prompts, incomplete self-correction.

### Independent Practice

- Purpose: Test learner production with minimal support.
- Suitable learning skills or stages: Stage 3 work updates, Stage 6 technical explanation, Stage 8 writing, Stage 10 long-form speaking.
- Expected learner output: spoken response, written message, technical explanation, listening summary, or interview answer.
- Evidence value: supports M4 when reviewed for correctness, independence, clarity, and task fit.
- Common failure patterns: grammar collapse under independence, vague output, direct translation, weak structure.

### Mixed Practice

- Purpose: Combine grammar, vocabulary, listening, speaking, writing, workplace context, and review.
- Suitable learning skills or stages: Stage 5 listening plus speaking, Stage 7 meetings, Stage 8 writing, Stage 10 fluency and nuance.
- Expected learner output: role-play summary, meeting response, blocker explanation, revised written artifact, or combined listening/speaking response.
- Evidence value: supports transfer when multiple learning skills are demonstrated together.
- Common failure patterns: success in one mode only, weak transfer, missing evidence for one component.

### Error-Based Practice

- Purpose: Repair observed grammar, pronunciation, phrasing, vocabulary, clarity, tone, listening, or structure mistakes.
- Suitable learning skills or stages: all stages, especially review and maintenance.
- Expected learner output: corrected sentence, revised message, repeated spoken answer, pronunciation repair, or self-correction note.
- Evidence value: supports evidence for weak-area repair when the correction is reviewed.
- Common failure patterns: correcting only surface form, introducing new errors, treating one correction as mastery.

### Project-Based Practice

- Purpose: Build speaking portfolio, interview portfolio, workplace communication portfolio, technical explanation portfolio, shadowing log, or writing portfolio artifacts.
- Suitable learning skills or stages: Stage 11 portfolio work and cross-cutting transfer learning skills.
- Expected learner output: reviewed artifact, recording summary, writing sample, interview response set, or technical explanation portfolio item.
- Evidence value: may support M5 when reviewed and connected to realistic communication contexts.
- Common failure patterns: artifact without review, artifact not mapped to learning skills, no reflection or limitation notes.

### Interview-Style Practice

- Purpose: Practice follow-up questions, technical explanations, behavioral answers, clarification, unknown questions, and pressure handling.
- Suitable learning skills or stages: Stage 9 interview English and Stage 10 explanation defense.
- Expected learner output: interview answer, follow-up response, clarification question, tradeoff defense, or self-corrected answer.
- Evidence value: may support M5 or M6 when responses are captured or summarized and evaluated.
- Common failure patterns: memorized answers, fluent but inaccurate responses, weak follow-up, pretending to know.

## Practice Difficulty Guidance

- D1: Recognition and controlled recall; learner identifies terms, grammar patterns, phrases, or communication structures.
- D2: Guided sentence or phrase production; learner produces controlled output with prompts or models.
- D3: Independent short output; learner produces short spoken or written output with minimal support.
- D4: Integrated workplace or technical communication; learner combines language skills in realistic contexts.
- D5: Portfolio, interview, or pressure-based defense; learner produces or defends output under review, follow-up, or realistic constraints.

Difficulty labels are practice guidance, not mastery records.

## Evidence-Producing Practice

Learner output can support evidence when it includes one or more of:

- Spoken answer captured or summarized.
- Written answer or revised message.
- Grammar correction before-and-after.
- Vocabulary or collocation used in context.
- Listening summary or detail extraction.
- Shadowing attempt with feedback.
- Pronunciation review note.
- Technical explanation.
- Meeting role-play summary.
- Interview answer with follow-up.
- Portfolio artifact.

Evidence SHOULD identify the learner output, target learning skill, feedback or review, limitations, and possible state impact.

Practice participation, passive listening, repeated shadowing, reading generated dialogues, or lesson completion alone does not imply mastery.

- Meaning in English for Software Engineers: learner can speak, write, listen, or respond without excessive prompting, translation, or scripted dependence.
- Evidence examples: free speaking response, independent written artifact, answer after minimal prompt, or self-managed clarification.

### Explanation Quality

- Meaning in English for Software Engineers: communication is clear, structured, relevant, detailed enough, and adapted to audience.
- Evidence examples: technical explanation, interview answer, blocker explanation, decision summary, or meeting contribution.

- Meaning in English for Software Engineers: learner uses English in new workplace, technical, interview, listening, writing, or portfolio contexts.

### Consistency

- Meaning in English for Software Engineers: learner performs reliably across repeated attempts and reduces recurring mistakes.
- Evidence examples: multiple reviewed outputs showing stable grammar, clarity, pronunciation, vocabulary use, or interaction ability.
- Warning signs: repeated same tense error, recurring direct translation, inconsistent pronunciation clarity, or unstable follow-up responses.

### Speed

- Meaning in English for Software Engineers: response time and fluency matter only when the context requires conversation, interviews, or time-sensitive communication.
- Evidence examples: timely answer with acceptable accuracy, smooth follow-up response, reduced hesitation, or clear status update under pressure.
- Warning signs: fast but inaccurate speech, fluent memorized answers, or speed used to ignore clarity and correctness.
